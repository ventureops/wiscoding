# 기여 가이드

wiscoding에 자료를 공유해주셔서 감사합니다. 이 문서는 어떤 글이 환영받고, 어떻게 제출하며, 어떤 형식으로 작성하면 좋은지 안내합니다.

## 환영하는 글

- 직접 써보고 동작을 확인한 **실전 팁**
- 한 가지 문제를 해결하는 **재현 가능한 절차**
- 시행착오 끝에 정리한 **워크플로**나 **프롬프트 패턴**
- 다른 사람이 따라할 수 있도록 단계별로 적은 **튜토리얼**

## 피해야 할 글

- **검증하지 않은 내용** — "들어보니 이렇다더라" 수준의 추측
- **민감정보가 포함된 글** — API 키, 토큰, 사내 코드, 실명, 사내 URL
- **단순 링크 모음** — 직접 정리한 요점 없이 외부 링크만 나열
- **광고성 글** — 특정 유료 서비스의 홍보가 주된 목적인 글
- **출처가 불분명한 번역·인용** — 원문 링크와 라이선스 확인이 안 된 글

## 제출 방법

세 가지 경로가 있습니다.

### 1. 이슈로 새 팁 제출 (가장 권장)

가장 부담 없는 방법입니다. 운영자가 검토 후 정식 문서로 편입합니다.

1. [새 이슈](../../issues/new/choose) 에서 **"새로운 팁"** 템플릿 선택
2. 양식에 맞춰 내용 작성 (제목, 분류, 상황, 방법, 주의사항, 검증환경, 출처)
3. 체크박스 3개 모두 확인 후 제출
4. 운영자 검토 → 의견 교환 → 정식 문서 편입 → 이슈 종료

운영자가 문서로 옮기는 과정에서 표현·구조를 다듬을 수 있습니다.

### 2. Discussions에서 질문·토론

확신이 없는 내용, 가벼운 질문, 의견 교환은 이슈 대신 [Discussions](../../discussions) 를 이용합니다.

- 답이 잘 정리되면 운영자가 정식 문서로 옮길 수 있습니다.
- "이거 맞나요?" 같은 검증 단계의 글에 적합합니다.

### 3. Pull Request

기존 문서의 오타·내용 수정·예제 보완은 PR을 직접 보내주세요.

1. 리포지토리 포크
2. `fix/오타-수정-문서명` 같은 명확한 브랜치명 사용
3. 한 PR에 한 가지 변경만 (가능하면)
4. 커밋 메시지는 한국어 또는 영어, 변경 의도를 한 문장으로
5. PR 본문에 **무엇을, 왜** 바꿨는지 적기

큰 폭의 신규 문서 추가는 PR보다 이슈를 먼저 권장합니다 (방향 합의 후 작성).

## 문서 작성 규칙

### 파일명

- 영문 소문자, 단어는 하이픈으로 구분: `claude-code-hooks-basics.md`
- 너무 일반적인 이름 금지: `tip.md` `note.md` ❌
- 주제가 드러나게: `mcp-server-setup-on-mac.md` ✅

### 위치

| 주제 | 위치 |
|---|---|
| Claude Code 자체 기능 | `docs/claude-code/` |
| AI 협업 워크플로·프롬프트 | `docs/vibe-coding/` |
| 프로젝트 운영·배포·협업 | `docs/project-ops/` |
| 외부 도구 연동 | `docs/tools/` |
| 재사용 템플릿 | `templates/` |

### 문서 구조

각 문서는 다음 구조를 따릅니다.

```markdown
# 제목 (무엇에 관한 글인지 한 줄)

> 한 줄 요약: 어떤 상황에서 이 글이 도움이 되는지

## 상황
어떤 문제·맥락에서 이 방법이 필요한지

## 방법
단계별로 따라할 수 있게. 명령어는 코드 블록으로.

## 주의사항
함정, 흔한 실수, 안 통하는 케이스

## 검증환경
- OS:
- 도구 버전:
- 마지막 확인일:

## 출처·참고
- 링크
```

### 형식

- **마크다운** 사용, GitHub Flavored Markdown 기준
- **코드 블록**은 언어 표기 (```bash, ```ts 등)
- **이미지**는 같은 폴더 `assets/` 하위에 두고 상대 경로로
- **줄 길이** 제한 없음 (적절히 끊어 읽기 쉽게)
- **문체** 자유 (반말·존댓말 어느 쪽이든 일관되게)

## 검토 기준

운영자는 다음 기준으로 검토합니다.

1. **재현 가능한가** — 적힌 대로 따라하면 같은 결과가 나오는가
2. **민감정보가 없는가** — 키·토큰·실명·사내 정보 누락 없이 제거됐는가
3. **출처가 명확한가** — 외부 자료 인용이 적절히 표기됐는가
4. **위치가 맞는가** — 폴더 분류가 적절한가
5. **중복이 없는가** — 기존 문서와 겹치지 않거나, 겹친다면 보완·통합

부족한 부분은 코멘트로 의견을 드리고, 함께 다듬은 뒤 편입합니다.

## 수정 정책

- **편입된 문서의 수정**: 누구나 PR로 제안 가능. 원작자에게 가능한 한 알립니다.
- **잘못된 정보 발견 시**: "문서 수정 요청" 이슈 템플릿 사용
- **삭제·통합**: 운영자 판단으로 진행. 가능한 한 사전에 코멘트로 안내합니다.
- **라이선스**: 모든 기여 내용은 자동으로 [CC BY 4.0](LICENSE) 으로 공유에 동의한 것으로 간주합니다.
