# Claude Code 슬래시 커맨드 만들기

> `~/.claude/commands/` 에 마크다운 파일 하나로 나만의 슬래시 커맨드를 만드는 방법.

## 상황

자주 반복하는 작업(정리, 요약, 변환 등)을 매번 긴 프롬프트로 입력하는 대신, `/이름` 한 줄로 실행하고 싶을 때.

**핵심 3줄 요약:**
- 슬래시 커맨드는 `~/.claude/commands/<이름>.md` 파일 하나로 만든다.
- **사용자가 명시적으로 호출**하는 프롬프트 단축키. 스킬(Skill)과는 다른 개념.
- `$ARGUMENTS`로 사용자 입력을 받는다.

## 방법

### 슬래시 커맨드 vs 스킬(Skill) — 먼저 알고 가기

| 구분 | 슬래시 커맨드 | 스킬 (Skill) |
|---|---|---|
| **위치** | `~/.claude/commands/*.md` | `~/.claude/skills/*/SKILL.md` (또는 플러그인) |
| **호출 방식** | 사용자가 `/이름 [인자]`로 **명시적** 호출 | Claude가 상황에 맞으면 **자동 발동** (또는 사용자 호출) |
| **구조** | 단일 마크다운 파일 | 폴더 + frontmatter + 추가 리소스 가능 |
| **발동 조건** | 사용자 입력 시 | `description`에 매칭되는 작업이 감지될 때 |
| **노출 위치** | `/` 입력 시 자동완성 목록 | 시스템 메시지의 available skills 목록 |
| **적합한 용도** | 정해진 작업을 빠르게 실행 | 특정 영역 전반의 행동 가이드·절차 정의 |

> 참고: Claude Code는 슬래시 커맨드도 Skill 도구로 호출 가능한 형태로 노출한다 (시스템 메시지의 skills 목록에 함께 보임). 호출 방법이 두 가지가 될 뿐, 본질은 슬래시 커맨드.

### 파일 구조

```
~/.claude/
├── commands/            ← 전역 (모든 프로젝트에서 사용 가능)
│   ├── my-command.md
│   └── another-command.md
└── skills/              ← 스킬 (별도 구조)

<프로젝트루트>/.claude/
└── commands/            ← 프로젝트 로컬 (해당 프로젝트에서만)
    └── project-command.md
```

### 커맨드 파일 작성법

**기본 형식:**

```markdown
---
description: 커맨드가 하는 일을 한 줄로 설명 (자동완성 목록에 표시됨)
argument-hint: <인자 설명>
---

# 커맨드 이름

대상 입력: `$ARGUMENTS`

## 절차
1. ...
2. ...
```

**핵심 요소:**

| 요소 | 설명 |
|---|---|
| `description` | frontmatter. `/` 입력 시 자동완성 목록에 표시되는 설명. |
| `argument-hint` | frontmatter. `/커맨드 <여기에 힌트>` 형태로 표시됨. |
| `$ARGUMENTS` | 사용자가 커맨드 뒤에 입력한 텍스트로 자동 치환됨. |

### 실전 예시: `/clipping-summarize` 커맨드

웹 클리핑 노트를 읽어 구조화된 레퍼런스 노트로 정리하는 커맨드.

**사용법:**
```
/clipping-summarize Clippings/기사제목.md    ← 경로 직접 지정
/clipping-summarize 키워드                   ← 키워드로 검색
/clipping-summarize                          ← 인자 없으면 대화로 물어봄
```

**커맨드 내부 절차 설계:**
1. `$ARGUMENTS` 해석 → 경로 / 키워드 / 빈값 분기
2. 키워드이면 검색으로 후보 찾기
3. 후보 분기:
   - 0건 → 재입력 요청
   - 1건 → 자동 진행
   - 2건 이상 → **번호 목록으로 제시, 사용자가 선택** (`1,3` 또는 `전체`/`all`)
4. 확정 파일들을 **병렬**로 읽기
5. 관련 노트 검색 (키워드 1–2개)
6. 정해진 템플릿으로 정리 노트 작성
7. 지정된 경로에 저장

### 커맨드 vs Claude Chat — 언제 쓸까

| 상황 | 더 유리한 도구 |
|---|---|
| 소스가 로컬 파일 | **Claude Code + 커맨드** (파일 직접 접근, 복붙 왕복 불필요) |
| 결과를 파일로 저장 | **Claude Code + 커맨드** |
| 반복 작업을 자동화 | **Claude Code + 커맨드** |
| 텍스트를 이미 손에 들고 있고 빠른 요약만 필요 | Claude Chat이 빠름 |

## 주의사항

커맨드를 설계하면서 배운 것들:

**다중 검색 결과 처리**
키워드 검색 결과가 여러 개일 때 Claude가 임의로 판단하면 엉뚱한 파일을 처리할 수 있다. **사용자에게 번호로 선택**하게 하는 것이 더 빠르고 정확하다.

**병렬 도구 호출 명시**
여러 파일을 읽을 때는 순차 실행 대신 **병렬 호출**을 커맨드에 명시해야 빠르다. "병렬로 읽어라"고 적어두면 Claude가 따른다.

**background + sleep 패턴 금지**
```bash
# 이렇게 하면 불필요하게 대기 시간 낭비
some-tool &
sleep 3
cat output_file
```
커맨드에 foreground 실행을 기본으로 명시한다.

**저장 경로 고정**
저장 폴더를 커맨드에 하드코딩하면 일관성이 생긴다. 변경할 때는 커맨드 파일만 수정하면 됨.

**덮어쓰기 방지**
동일 파일명이 이미 있을 때 자동 덮어쓰기 방지 로직을 명시해두면 실수를 막을 수 있다.

**속도 팁**
- **Opus**는 품질 최고, 속도 느림. 정형화 작업엔 과함.
- **Sonnet**은 2–3배 빠르고 이 종류 작업엔 품질 충분. `/model sonnet`으로 전환.
- **`/fast`** 모드: Opus 4.6을 빠른 출력 모드로. Opus 품질 유지하면서 속도 개선.
- 커맨드 안에서 도구 왕복(검색 → 읽기 → 검색 → 쓰기)은 누적 지연이 생기므로, 검색 단계를 사용자가 미리 줄여주면 (정확한 경로 직접 입력) 가장 빠르다.
