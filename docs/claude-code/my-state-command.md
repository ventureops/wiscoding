# 클로드 환경 인벤토리 자동화 — `/my-state` 만들기

> 메모리·스킬·슬래시 명령·hooks·플러그인 등 누적되는 "숨은 상태"를 한 줄 명령으로 점검하는 자가 인벤토리 슬래시 명령을 만든 과정과 설계 원칙.

## 상황

Claude Code를 오래 쓸수록 **숨은 상태(hidden state)** 가 누적된다.

- "예전에 이 폴더에서 자동 적용되는 사용자 메모리를 저장했었지... 어디 있더라?"
- "스킬 만들었는데 description이 뭐였더라?"
- "이 settings에 hook 걸어둔 게 있었나?"

전부 어딘가에 **파일로 존재**하지만, 위치와 내용을 머리로 추적하는 순간 인지 부하가 폭증한다. 메모리는 작업 디렉토리별로 분리되어 더 헷갈린다.

**핵심 통찰**: 영구 설정의 진짜 병목은 **"기억해야 한다는 강박"** 이다. 기억하지 말고, 필요할 때 보면 된다. `/my-state` 한 줄로 인벤토리화하면 인지 부하 자체가 사라진다.

## 방법

### 클로드 코드의 영구 상태 카테고리 (10가지)

`/my-state`가 커버해야 할 영역. 사용자가 직간접적으로 만드는 영구 상태:

| 종류 | 위치 | 무엇을 바꾸나 |
|---|---|---|
| **메모리** | `~/.claude/projects/<인코딩폴더>/memory/` | 사용자/프로젝트 맥락 (자동 인입) |
| **스킬** | `~/.claude/skills/` 또는 프로젝트 `.claude/skills/` | 자동 발동 절차 |
| **슬래시 명령** | `~/.claude/commands/` 또는 프로젝트 `.claude/commands/` | 단축 명령 |
| **CLAUDE.md** | `~/.claude/CLAUDE.md` (전역) 또는 프로젝트 루트 | 항상 적용되는 시스템 프롬프트 |
| **settings.json** | `~/.claude/settings.json`, `.claude/settings.json` | 모델, 권한, 환경변수, hooks |
| **hooks** | settings.json 내부 | 특정 이벤트 시 자동 실행되는 명령 |
| **권한 허용목록** | settings.json | 자동 승인되는 도구/명령 |
| **MCP 서버** | `~/.claude/mcp.json` 등 | 연결된 외부 서비스 |
| **플러그인** | 마켓플레이스 설치 | 통째로 묶인 스킬/명령 세트 |
| **서브에이전트** | `~/.claude/agents/` | 특정 목적 미니 에이전트 |

→ 사용자가 의식해야 할 카테고리만 **10개**. 외울 게 아니라 `/my-state`가 항상 보여주게 만드는 게 정답.

### 만든 절차 — 4단계 워크플로우

**Step 1. 슬래시 명령 파일 생성**
- `~/.claude/commands/my-state.md` 생성 (전역 적용)
- frontmatter에 `description`, `allowed-tools: Bash, Read` 명시

**Step 2. 절차 작성 — "수집 → 출력" 2단 구조**
- **수집**: 병렬 Bash로 프로젝트 `.claude/`, 전역 `~/.claude/`, 메모리 디렉토리, settings.json 키, 플러그인을 한 번에 가져오기
- **출력**: 카테고리별 고정 형식으로 한 화면 인벤토리

**Step 3. 출력 원칙 명시 (인지 부하 차단의 핵심)**
- **빈 카테고리도 항상 표시** — "이 카테고리도 있다"는 인지 자체를 만듦
- **각 항목 description 자동 추출** — frontmatter에서 가져옴, 50자 절단
- **사족 금지** — 인벤토리만, 해석은 사용자 몫

**Step 4. 1회 실제 사용 → 보강**
- 첫 실행에서 발견된 이슈 메모 (중복 표시, 긴 description)
- 룰 추가 → 다시 검증

→ 슬래시 명령 파일 작성 자체는 [Claude Code 슬래시 커맨드 만들기](custom-slash-commands.md) 참고.

### 핵심 설계 원칙 4가지

1. **빈 카테고리도 표시한다** — `(없음)` 명시. "이게 있다는 사실 자체"가 사용자에게 인지 자료가 됨. 비어 있으면 안 보여주는 방식은 사용자에게 "이 카테고리가 있는지조차 모르게" 만든다.
2. **description은 frontmatter에서 자동 추출** — 사용자가 description을 잘 적어두면 인벤토리 품질도 자동으로 올라간다. 좋은 메타데이터의 복리 효과.
3. **사족 금지** — "정리해드렸습니다" 같은 메타코멘트 0. 인벤토리는 데이터지 보고서가 아니다.
4. **카테고리 순서 = 인지 우선순위** — 현재 위치 → 프로젝트 → 메모리 → 전역 → 플러그인. 가장 가까운 맥락부터 보여줌.

### 출력 형식 예시

```
📁 현재 프로젝트
   /Users/<이름>/path/to/your/project

━━━ 프로젝트 (.claude/) ━━━
   CLAUDE.md      : (없음)
   settings.json  : (없음)
   .local.json    : (없음)
   사용자 자원    : 0개

━━━ 메모리 (이 프로젝트 한정) ━━━
   1개
     - user_workflow_setup.md — User's chosen Claude Code workflow…

━━━ 전역 (~/.claude/) ━━━
   CLAUDE.md      : (없음)
   settings.json  : permissions(allow 53개), hooks 0개
   사용자 자원    : 총 4개 (스킬 1 / 명령 4 / 동명 1)
     [스킬+명령] save-md            — 방금 대화 내용을 Obsidian 볼트에 마크다운…
     [명령]      clipping-summarize — Clippings 노트를 30 Tools에 정리
     [명령]      my-state           — 현재 클로드 환경 인벤토리 한눈에 보기
     [명령]      research           — Super Research Agent (할루시네이션 없는…
   에이전트       : 0개

━━━ 플러그인 ━━━
   3개 활성
     - vercel@claude-plugins-official
     - superpowers@claude-plugins-official
     - obsidian@obsidian-skills
```

### 1차 사용 후 보강 — 병합/절단/정렬 룰

**발견된 이슈**: 첫 실행에서 동명의 스킬+명령(예: `save-md`)이 두 줄로 중복 표시. description이 50자를 넘어 두 줄로 표시되어 가독성 저하.

**추가된 룰**

| 룰 | 효과 |
|---|---|
| 동명 스킬+명령 병합 | `[스킬+명령]` 한 줄로 표시. description 충돌 시 명령 쪽 우선 (보통 더 짧음) |
| 50자 무조건 절단 | `…` 추가, 한 줄 보장, 시각 길이 일관성 확보 |
| description 정규화 | 앞뒤 따옴표 제거, 줄바꿈 제거 |
| 카테고리 카운트 분해 | "총 4개 (스킬 1 / 명령 4 / 동명 1)" — 분포 한눈에 파악 |
| 정렬 규칙 | `[스킬+명령]` → `[스킬]` → `[명령]`, 같은 태그 내 알파벳순 |

## 주의사항

### 권장 호출 시점

- **새 세션 시작 시 한 번** — 현재 환경 파악
- **새 스킬/명령/메모리 추가 직후** — 변경 확인
- **월 1회 정기 점검** — 불필요한 자원 정리 기회

### 한계 — `/my-state`가 못 보여주는 것

- **왜 이걸 만들었는지**의 맥락 (의도)
- **언제 만들었는지**의 이력
- **어떻게 진화했는지**의 변화 추적

### 보완: 인덱스 노트로 의도·이력 별도 관리

별도의 인덱스 노트(예: 옵시디언 또는 별도 파일)를 하나 만들고:
- 새 자원 추가 시 한 줄씩 적기 (날짜, 이름, 의도)
- `/my-state`가 **현재 상태**, 인덱스 노트가 **의도와 이력** — 둘이 보완

### 다음 단계 아이디어

- `/my-state diff` — 마지막 호출 대비 변경된 항목만 표시 (변경 추적)
- `/my-state plugins` — 플러그인이 제공하는 스킬/명령 목록까지 깊이 있게 표시
- statusline 통합 — VS Code statusline에 "메모리 N개, 스킬 M개" 항상 표시
- 세션 시작 자동 호출 — hook으로 새 세션마다 인벤토리 자동 표시 (단, 항상 띄우면 부담스러울 수 있음 → 옵션화)
